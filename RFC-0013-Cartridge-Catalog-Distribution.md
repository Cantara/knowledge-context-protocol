## What This RFC Proposes

A `catalog.yaml` format that lets developers declare, version, and distribute KCP
manifests as static artifacts — independently of any running service.

The catalog answers: *"I have a set of `knowledge.yaml` files. How do I share them
with my team, install them on a new machine, and know when they're stale?"*

All fields are optional. All additions are backward compatible. Existing manifests
require no changes.

---

## The Problem

### Federation is the runtime plane. Distribution is the authoring plane.

RFC-0003 and RFC-0011 define how a **running agent** traverses live KCP servers to
discover manifests. This works well once an agent is connected to a known entry point.

It does not address the earlier problem: **how does a developer share a `knowledge.yaml`
file with a teammate or a second machine?** A manifest sitting in a local directory
after being generated by `synthesis export --format kcp` or
[kcp-triage](https://github.com/StigLau/kcp-triage) has no specified path to any
other developer's machine.

Today, the only answer is "copy the file manually." The KCP ecosystem is acquiring
tooling that generates manifests automatically. That tooling needs a distribution path.

### Generated manifests accumulate without lifecycle management

Automated tools (Synthesis, kcp-triage, OpenAPI-to-KCP converters) generate manifests
from external observations. These manifests:

- Have no stable location other than the machine that generated them
- Have no versioning contract with the source they describe
- Have no mechanism to express "this manifest depends on that one being present"
- Have no way for a consumer to know whether the manifest is stale relative to its source

A developer who generates 20 manifests across 5 codebases and wants to share them with
3 teammates currently has no KCP-native way to do so.

### No dependency declaration between cartridges

A manifest for `service-api` may assume that the consuming agent already holds a
manifest for `service-auth` (shared authentication patterns, token formats, base URLs).
There is currently no way to express this dependency in KCP. The consuming agent either
discovers the dependency at runtime — if both are hosted as live endpoints and federated
— or silently operates without the context it needs.

For teams that distribute manifests as static files — checked into repos, not served
from live servers — this runtime federation path does not exist. The dependency must be
resolvable at install time.

## Design

### `catalog.yaml`

A `catalog.yaml` file declares a named collection of KCP manifest references. Each
entry names a manifest, identifies where to fetch it, records its version and
provenance, and optionally declares dependencies on other catalog entries.

```yaml
kcp_catalog_version: "0.1"

catalog:
  name: myproject-cartridges
  description: "KCP manifests for the MyProject platform stack"
  maintainer: platform-team@myproject.example

entries:
  - name: myproject-internal-api
    source: https://raw.githubusercontent.com/myorg/kcps/main/api/knowledge.yaml
    version: 0.4.1
    generated: "2026-03-20"
    source_commit: a3f91bc
    depends_on:
      - myproject-auth

  - name: myproject-auth
    source: ./cartridges/myproject-auth/knowledge.yaml
    version: 0.2.0
    generated: "2026-03-15"

  - name: nis2-compliance
    source: https://raw.githubusercontent.com/example-org/kcps/main/nis2/knowledge.yaml
    version: 1.0.0
```

### Entry field reference

| Field | Required | Type | Description |
|-------|----------|------|-------------|
| `name` | REQUIRED | string | Unique identifier for this manifest within the catalog. Used for dependency references. |
| `source` | REQUIRED | string | Where to fetch the manifest. Supports local paths, raw HTTPS URLs, and git repository references. |
| `version` | RECOMMENDED | semver string | The declared version of the manifest at the time it was added to the catalog. Compared using semver as defined by [SemVer 2.0.0](https://semver.org/). |
| `generated` | OPTIONAL | ISO 8601 date | When this manifest was generated or last verified against its source. Enables staleness detection. |
| `source_commit` | OPTIONAL | string | Git commit SHA of the source repository at generation time. Enables precise staleness detection when the source is a git repo. |
| `depends_on` | OPTIONAL | list of `name` strings | Names of other catalog entries that MUST be installed before this manifest is useful. References MUST be catalog-local; cross-catalog dependencies are out of scope. |

### `source` URL types

| Pattern | Example | Behaviour |
|---------|---------|-----------|
| Local path (relative) | `./cartridges/auth/knowledge.yaml` | Resolved relative to the `catalog.yaml` file location. |
| Local path (absolute) | `/home/user/.kcp/cartridges/auth/knowledge.yaml` | Absolute path on the local filesystem. |
| Raw HTTPS | `https://raw.githubusercontent.com/org/repo/main/kcp/knowledge.yaml` | Fetched directly. No authentication beyond what the HTTP client provides. |
| Git repository | `git+https://github.com/org/repo.git//path/to/knowledge.yaml@v0.4.1` | Fetched from a specific git ref. Grammar defined below. |

**Git source URL grammar:**

```
git-source  = "git+" https-url "//" file-path [ "@" ref ]
https-url   = "https://" host "/" repo-path ".git"
file-path   = path to knowledge.yaml within the repository
ref         = branch-name | tag-name | full-commit-sha
```

- The `//` separator delimits the repository URL from the file path within the repository.
- `@ref` is OPTIONAL. If absent, the implementation MUST use the repository's default branch.
- `ref` MAY be a branch name, tag name, or 40-character full commit SHA. Short SHAs are NOT RECOMMENDED due to ambiguity.

Example: `git+https://github.com/acme/kcps.git//api/knowledge.yaml@v0.4.1`

Implementations MUST support local paths and raw HTTPS URLs. Git repository sources are
RECOMMENDED for production use. Unknown source schemes MUST be reported as errors, not
silently ignored.

### Dependency resolution

When installing a catalog entry that declares `depends_on`, all named dependencies MUST be
installed first. Circular dependencies MUST be detected and reported as an error.

```yaml
# myproject-auth is installed before myproject-internal-api.
entries:
  - name: myproject-internal-api
    source: ...
    depends_on:
      - myproject-auth

  - name: myproject-auth
    source: ...
```

`depends_on` entries MUST reference `name` values declared elsewhere in the same catalog.
Cross-catalog dependencies are out of scope.

### Catalog location convention

A `catalog.yaml` SHOULD be placed at one of the following well-known locations,
in order of preference:

1. `.kcp/catalog.yaml` — project-local, co-located with source code
2. `~/.kcp/catalog.yaml` — user-global, for practitioner rigs
3. Any path passed explicitly via tooling

A project-local catalog committed to version control gives every team member the same
manifest set with a single install step.

### Staleness detection

An entry is considered **stale** when any of the following conditions hold:

- `source_commit` is set and the source repository's current HEAD for the relevant
  path differs from `source_commit`.
- `generated` is set and the source file's modification time is newer than `generated`.
- The installed manifest's `version` field (from SPEC.md §2) is lower than the
  `version` declared in the catalog entry, as determined by semver comparison
  per [SemVer 2.0.0](https://semver.org/).

Implementations SHOULD surface a staleness warning. They MUST NOT refuse to use a
stale manifest — the operator decides whether to update.

## Relationship to RFC-0003 and RFC-0011 (Federation)

Federation (RFC-0003, RFC-0011) operates at **runtime**: a running agent traverses
live KCP servers using HATEOAS links in `manifests[]` blocks.

Distribution (this RFC) operates at **authoring time**: a developer or CI pipeline
fetches manifests from source and places them where agents can read them.

The two planes are complementary:

- A manifest installed via a catalog can also declare `manifests[]` federation links
  — the catalog gets it there; federation takes over from there.
- A live KCP server may serve a manifest whose `catalog.yaml` entry points back to
  the server's canonical URL. Distribution is the write path; federation is the read
  path.

There is no conflict. A manifest can be distributed and federated simultaneously.

## Relationship to RFC-0012 (Capability Discovery Provenance)

RFC-0012 `discovery` blocks record how each unit was found and how confident the
discoverer was. RFC-0013 `generated` and `source_commit` fields record when and from
what source the **manifest as a whole** was generated.

Together they provide full provenance at two levels:

```yaml
# In catalog.yaml — manifest-level provenance (RFC-0013)
entries:
  - name: acme-hr-portal
    source: https://raw.githubusercontent.com/acme/kcps/main/hr/knowledge.yaml
    version: 0.3.0
    generated: "2026-03-10"
    source_commit: a1b2c3d
```

```yaml
# In knowledge.yaml — unit-level provenance (RFC-0012)
units:
  - id: submit-expense-report
    discovery:
      source: web_traversal
      confidence: 0.72
      observed_at: "2026-03-10T14:22:00Z"
```

The catalog tells you *which version of the manifest to use and where it came from*.
The `discovery` block tells you *how confidently any given unit within it was
established*.

## Conformance

| Feature | Level | Notes |
|---------|-------|-------|
| `catalog.yaml` parsing | Level 1 | Tooling that installs manifests from a catalog MUST implement this. |
| Local path source resolution | Level 1 | MUST be supported. |
| Raw HTTPS source fetch | Level 1 | MUST be supported. |
| `depends_on` dependency resolution | Level 1 | MUST detect and report circular dependencies. |
| Staleness detection via `version` | Level 2 | RECOMMENDED. |
| Staleness detection via `source_commit` | Level 2 | RECOMMENDED for git-backed sources. |
| Git repository sources | Level 2 | RECOMMENDED for production catalogs. |

## Backward Compatibility

`catalog.yaml` is a new file format. It does not modify `knowledge.yaml` or `SPEC.md`.
No existing manifest is affected.

The `version` field on `knowledge.yaml` root (SPEC.md §2) is already defined.
This RFC does not change its semantics — it adds the convention that catalog entries
SHOULD mirror this version for staleness comparison.

---

## Decisions Made During Review

**Cross-catalog dependencies:** Out of scope for this RFC. A `depends_on` entry MUST
reference a name within the same catalog. Cross-catalog dependency resolution implies a
hosted registry and belongs in a future RFC.

**`registry:` field:** Out of scope. Named registry lookup implies a hosted registry
service. Deferred to a future RFC.

**Lock file (`catalog.lock`):** Tooling-level, not spec-level. A lock file recording
exact resolved sources and commit SHAs at install time is an implementation artifact.
Two implementations with different lock file formats are still interoperable if they
both parse `catalog.yaml` correctly. No spec entry required.

## Relationship to Other RFCs

- **RFC-0003 (Federation):** Federation is the runtime read path. Distribution is the authoring-time write path. They compose without conflict.
- **RFC-0011 (Org Federation):** Org federation assumes agents are already connected to a known hub. Catalogs distribute manifests to machines that have no hub connection yet — the pre-federation bootstrap.
- **RFC-0012 (Capability Discovery Provenance):** `discovery` on units records per-capability confidence. `generated` and `source_commit` on catalog entries record manifest-level provenance. Together: full two-level provenance from manifest origin to individual capability certainty.
- **RFC-0008 (Agent Readiness / llms.txt):** `llms.txt` bootstraps an agent's first manifest lookup from a domain name. A catalog entry can reference the manifest discovered via `llms.txt`, grounding runtime bootstrap in a versioned static artifact.
