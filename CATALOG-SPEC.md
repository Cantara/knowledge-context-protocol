# KCP Catalog Specification

**Version:** 0.1
**Status:** Draft
**Date:** 2026-03-25
**Repository:** github.com/cantara/knowledge-context-protocol
**Parent spec:** [SPEC.md](./SPEC.md) (KCP v0.13)

---

## Abstract

The KCP Catalog Specification defines the `catalog.yaml` file format for declaring,
versioning, and distributing KCP manifests (`knowledge.yaml` files) as static artifacts,
independently of any running service.

A catalog answers: *"I have a set of `knowledge.yaml` files. How do I share them with my
team, install them on a new machine, and know when they are stale?"*

The catalog format is a companion to the KCP specification. It does not modify
`knowledge.yaml`. Every manifest referenced by a catalog entry MUST conform to
[SPEC.md](./SPEC.md).

---

## Status of This Document

This is a draft specification. The format is intentionally minimal and subject to revision
based on implementation feedback. Implementations SHOULD declare which version of this
specification they conform to using the `catalog_version` field.

## 1. Background: Distribution vs Federation

The KCP specification (SPEC.md §3.6, RFC-0003, RFC-0011) defines how a **running agent**
traverses live KCP servers to discover and federate manifests. This is the **runtime plane**.

The catalog format addresses the **authoring plane**: how a developer shares a
`knowledge.yaml` file with a teammate or a second machine before any agent is involved.
A manifest sitting on a local filesystem after being generated by automated tooling has no
specified distribution path without a catalog.

These planes are complementary and composable:

- **Distribution** (this spec) is the write path — a developer fetches manifests from
  source and places them where agents can read them.
- **Federation** (SPEC.md §3.6) is the read path — a running agent traverses live KCP
  servers using `manifests[]` federation links.

A manifest installed via a catalog MAY also declare `manifests[]` federation links. There
is no conflict. A manifest can be distributed and federated simultaneously.

---

## 2. File Format

A `catalog.yaml` file MUST be valid YAML 1.2 and UTF-8 encoded without a BOM.

Parsers MUST silently ignore fields they do not recognise. This ensures forward
compatibility as the catalog format evolves.

## 3. Root Structure

```yaml
catalog_version: "0.1"

catalog:
  name: myproject-cartridges
  description: "KCP manifests for the MyProject platform stack"
  maintainer: platform-team@myproject.example

entries:
  - name: myproject-internal-api
    source: https://raw.githubusercontent.com/myorg/kcps/main/api/knowledge.yaml
    version: 0.4.1
    generated: "2026-03-20"
    source_commit: a3f91bc
    depends_on:
      - myproject-auth

  - name: myproject-auth
    source: ./cartridges/myproject-auth/knowledge.yaml
    version: 0.2.0
    generated: "2026-03-15"

  - name: nis2-compliance
    source: https://raw.githubusercontent.com/example-org/kcps/main/nis2/knowledge.yaml
    version: 1.0.0
```

### Root fields

| Field | Required | Type | Description |
|-------|----------|------|-------------|
| `catalog_version` | REQUIRED | string | The catalog specification version. MUST be `"0.1"` for this version. |
| `catalog` | OPTIONAL | object | Metadata block describing the catalog itself. |
| `catalog.name` | OPTIONAL | string | Human-readable name for the catalog. |
| `catalog.description` | OPTIONAL | string | Brief description of the manifest collection. |
| `catalog.maintainer` | OPTIONAL | string | Contact address or identifier for the catalog maintainer. |
| `entries` | REQUIRED | list | One or more manifest entries. MUST contain at least one entry. |

---

## 4. Entry Structure

Each entry in the `entries` list describes one KCP manifest.

### Entry fields

| Field | Required | Type | Description |
|-------|----------|------|-------------|
| `name` | REQUIRED | string | Unique identifier for this manifest within the catalog. Used in `depends_on` references. MUST match `[a-z0-9][a-z0-9\-]*`. |
| `source` | REQUIRED | string | Where to fetch the manifest. See §5 for supported source types and grammar. |
| `version` | RECOMMENDED | semver string | The declared version of the manifest at the time it was added to the catalog. Compared using [SemVer 2.0.0](https://semver.org/). |
| `generated` | OPTIONAL | ISO 8601 date | When this manifest was generated or last verified against its source (e.g. `"2026-03-20"`). Enables staleness detection. |
| `source_commit` | OPTIONAL | string | Full 40-character git commit SHA of the source repository at generation time. Enables precise staleness detection. Short SHAs are NOT RECOMMENDED. |
| `depends_on` | OPTIONAL | list of strings | Names of other entries in this catalog that MUST be installed before this entry. See §6. |

---

## 5. Source Types

The `source` field identifies where to fetch the `knowledge.yaml` file.

### 5.1 Local path (relative)

A path beginning with `./` or `../` is resolved relative to the `catalog.yaml` file location.

```yaml
source: ./cartridges/auth/knowledge.yaml
```

### 5.2 Local path (absolute)

A path beginning with `/` is treated as an absolute filesystem path.

```yaml
source: /home/user/.kcp/cartridges/auth/knowledge.yaml
```

### 5.3 Raw HTTPS URL

Any URL beginning with `https://` is fetched directly via HTTP GET.

```yaml
source: https://raw.githubusercontent.com/org/repo/main/kcp/knowledge.yaml
```

No authentication is specified at this version. Implementations MAY support
environment-variable-based credentials (e.g. `GITHUB_TOKEN` for private repositories).

### 5.4 Git repository source

A URL beginning with `git+https://` identifies a file within a git repository at a
specific ref.

**Grammar:**

```
git-source  = "git+" https-url "//" file-path [ "@" ref ]
https-url   = "https://" host "/" repo-path ".git"
file-path   = 1*( ALPHA / DIGIT / "/" / "-" / "_" / "." )
ref         = branch-name / tag-name / commit-sha
commit-sha  = 40HEXDIG
```

- The `//` separator delimits the repository URL from the file path within the repository.
- `@ref` is OPTIONAL. If absent, the implementation MUST use the repository's default
  branch.
- `ref` MAY be a branch name, tag name, or 40-character full commit SHA.
  Short SHAs are NOT RECOMMENDED due to collision risk.

**Example:**

```yaml
source: git+https://github.com/acme/kcps.git//api/knowledge.yaml@v0.4.1
```

### 5.5 Supported source types (conformance)

| Source type | Support level |
|-------------|---------------|
| Local path (relative or absolute) | MUST be supported |
| Raw HTTPS URL | MUST be supported |
| Git repository source | RECOMMENDED |

Unknown source schemes MUST be reported as an error. Implementations MUST NOT silently
skip unknown schemes.

## 6. Dependency Resolution

An entry MAY declare `depends_on` — a list of `name` values identifying other entries
in the same catalog that MUST be installed before this entry.

```yaml
entries:
  - name: myproject-internal-api
    source: ...
    depends_on:
      - myproject-auth       # installed first

  - name: myproject-auth
    source: ...
```

### Normative rules

- `depends_on` values MUST reference `name` values declared elsewhere in the same catalog.
  A reference to an unknown name MUST be reported as an error.
- Cross-catalog dependencies (referencing entries in an external catalog) are not supported
  in this version.
- Circular dependencies (A → B → A) MUST be detected and reported as an error before any
  installation begins.
- Installation order MUST satisfy all `depends_on` constraints. Implementations MAY install
  independent entries in parallel.

---

## 7. Catalog Location Convention

A `catalog.yaml` SHOULD be placed at one of the following well-known locations,
in order of preference:

| Priority | Location | Scope |
|----------|----------|-------|
| 1 | `.kcp/catalog.yaml` | Project-local, committed to version control |
| 2 | `~/.kcp/catalog.yaml` | User-global, for practitioner rigs |
| 3 | Explicit path via tooling | Any path passed via CLI flag or config |

A project-local catalog committed to version control gives every team member an identical
manifest set. A `git clone` followed by a catalog install step is sufficient to reproduce
the full knowledge context on any machine.

---

## 8. Staleness Detection

An entry is considered **stale** when any of the following conditions hold:

1. `source_commit` is declared and the source repository's current HEAD for the relevant
   file path differs from `source_commit`.
2. `generated` is declared and the source file's last-modified timestamp is newer than
   `generated`.
3. The installed manifest's `version` field (SPEC.md §2) is lower than the `version`
   declared in the catalog entry, as determined by semver comparison per
   [SemVer 2.0.0](https://semver.org/).

Implementations SHOULD surface a staleness warning when any condition is met.
Implementations MUST NOT refuse to use or load a stale manifest — the operator
decides whether to update.

---

## 9. Relationship to KCP Federation

Catalog-based distribution and KCP federation (SPEC.md §3.6, RFC-0003, RFC-0011) operate
at different layers and compose without conflict.

| Dimension | Catalog (this spec) | Federation (SPEC.md §3.6) |
|-----------|---------------------|--------------------------|
| When | Authoring time | Runtime |
| Actor | Developer / CI pipeline | Running agent |
| Mechanism | Static file fetch from source | HATEOAS traversal of live servers |
| Direction | Write path — get manifests to machines | Read path — agent discovers context |

A manifest installed via a catalog MAY declare `manifests[]` federation links. The catalog
delivers the manifest to the agent's machine; federation extends discovery from there.

## 10. Relationship to RFC-0012 (Capability Discovery Provenance)

RFC-0012 `discovery` blocks on units record how each capability was found and how confident
the discoverer was (unit-level provenance). Catalog `generated` and `source_commit` fields
record when and from what source the manifest as a whole was generated (manifest-level
provenance).

Together they provide two-level provenance:

```yaml
# catalog.yaml — manifest-level provenance
entries:
  - name: acme-hr-portal
    source: https://raw.githubusercontent.com/acme/kcps/main/hr/knowledge.yaml
    version: 0.3.0
    generated: "2026-03-10"
    source_commit: a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2
```

```yaml
# knowledge.yaml — unit-level provenance (RFC-0012)
units:
  - id: submit-expense-report
    discovery:
      source: web_traversal
      confidence: 0.72
      observed_at: "2026-03-10T14:22:00Z"
```

## 11. Conformance Levels

| Feature | Level | Notes |
|---------|-------|-------|
| `catalog.yaml` parsing | Level 1 | MUST parse all REQUIRED and RECOMMENDED fields. |
| Local path source resolution | Level 1 | MUST support relative and absolute local paths. |
| Raw HTTPS source fetch | Level 1 | MUST support `https://` sources. |
| `depends_on` resolution | Level 1 | MUST detect circular deps. MUST error on unknown names. |
| Staleness detection via `version` | Level 2 | RECOMMENDED. Semver comparison per SemVer 2.0.0. |
| Staleness detection via `source_commit` | Level 2 | RECOMMENDED for git-backed sources. |
| Git repository source (`git+https://`) | Level 2 | RECOMMENDED for production catalogs. |
